# 📝 EduNote 기여하기 가이드

EduNote 프로젝트에 기여하기 위한 가이드입니다.
팀원이나 포트폴리오를 확인하는 분들이 쉽게 이해할 수 있도록 작성했습니다.

---

## 🚀 1. 브랜치 전략 (Branch Strategy)

### 🌳 브랜치 구조

**브랜치 이름 형식: `작업타입/영역/#이슈번호`**
> 예: `feat/ai/#12`, `fix/back/#25`

-   `main` : 릴리즈/배포용 브랜치
-   `dev` : 개발 통합 브랜치
    -   `feat/ai/#이슈번호` (AI 신규 기능)
    -   `feat/back/#이슈번호` (백엔드 신규 기능)
    -   `feat/front/#이슈번호` (프론트엔드 신규 기능)
    -   `fix/ai/#이슈번호` (AI 버그 수정)
    -   `fix/back/#이슈번호` (백엔드 버그 수정)
    -   `fix/front/#이슈번호` (프론트엔드 버그 수정)
    -   `docs/*` (문서 수정)
    -   `hotfix/*` (긴급 버그 수정)

### 🏷️ 네이밍 컨벤션

#### 1. 작업 타입 (Type)

* `feat`: 새로운 기능 개발
* `fix`: 버그 수정
* `hotfix`: 배포 후 긴급 버그 수정
* `build`: 빌드 관련 수정
* `chore`: 자잘한 수정 (설정, 패키지 설치 등)
* `ci`: CI 설정 수정
* `docs`: 문서 수정
* `style`: 코드 스타일/포맷 수정 (기능 변경 없음)
* `refactor`: 코드 리팩토링
* `test`: 테스트 코드 수정
* `perf`: 성능 개선

#### 2. 영역 (Scope)

* `back`: 백엔드 영역
* `front`: 프론트엔드 영역
* `ai`: AI 영역

### 🌊 브랜치 흐름 (Flow)

1.  모든 기능 브랜치(`feat`, `fix` 등)는 `dev` 브랜치에서 생성합니다.
2.  기능 개발이 완료되면, `dev` 브랜치를 대상으로 **Pull Request (PR)**를 생성합니다.
3.  코드 리뷰 및 테스트가 완료되면 `dev` 브랜치에 병합(Merge)합니다.
4.  `dev` 브랜치에서 충분히 안정화가 되면 `main` 브랜치로 병합하여 배포합니다.
5.  **[긴급]** 배포 후 긴급 버그는 `hotfix/*` 브랜치를 `main`에서 직접 생성하여 작업 후, `main`에 즉시 병합/배포하고, `dev` 브랜치에도 병합합니다.

## 💬 2. 커밋 & PR 규칙

### ✍️ 커밋 메시지

**형식: `타입: 간단 설명`**

| 타입 | 의미 | 예시 |
| :--- | :--- | :--- |
| **feat** | 새로운 기능 | `feat: AI 노트 요약 기능 추가` |
| **fix** | 버그 수정 | `fix: 로그인 시 404 오류 수정` |
| **docs** | 문서 관련 | `docs: API 명세서 v1.1 업데이트` |
| **chore** | 기타 변경 | `chore: 불필요한 .log 파일 삭제` |
| **refactor** | 리팩토링 | `refactor: PostService 로직 개선` |

### ✅ PR (Pull Request) 작성

* 모든 PR은 **`dev` 브랜치**를 대상으로 생성하는 것을 원칙으로 합니다.
* PR 제목 형식은 커밋 메시지 형식을 따릅니다: `[타입] 간단 설명`

> **PR 제목 예시:**
> * `[Feat] AI 요약 기능 및 API 추가`
> * `[Fix] 소셜 로그인 시 리다이렉트 오류 수정`
> * `[Hotfix] 긴급 배포용 로그인 오류 수정` (이 경우 `main` 대상)

# 📝 CONTRIBUTING Guide

EduNote 프로젝트에 기여하기 위한 가이드입니다.  
팀원이나 포폴용 프로젝트를 보는 사람이 쉽게 이해할 수 있도록 작성했습니다.

---

## 1️⃣ 브랜치 전략

### 배포용 브랜치
- `main`: 안정적인 배포용 브랜치

### 개발 통합 브랜치
- `dev`: 여러 기능을 통합하는 개발 브랜치

### 기능 개발 브랜치
- `feat/back/<기능명>`: 백엔드 기능 개발
- `feat/ai/<기능명>`: AI 기능 개발
- `feat/front/<기능명>`: 프론트엔드 기능 개발

### 배포 후 긴급 수정 브랜치
- `hotfix/back/<기능명>`: 백엔드 긴급 수정
- `hotfix/ai/<기능명>`: AI 긴급 수정
- `hotfix/front/<기능명>`: 프론트 긴급 수정

### 문서/설계 브랜치
- `docs/*`: API 명세, ERD, 가이드 등 문서 작성

---

### 🔹 브랜치 흐름

1. 모든 기능 브랜치는 `dev`에서 파생  
2. 기능 개발 완료 후 PR → `dev` 병합 → 충분히 테스트  
3. 안정화 후 → `main` 병합 → 배포  
4. 배포 후 긴급 버그 → `hotfix/*` 브랜치 생성 → main 바로 배포 → dev 병합  
5. 문서 작업 → `docs/*` 브랜치 생성 → dev/main 병합

---

## 2️⃣ PR & 커밋 규칙

### 커밋 메시지
| 타입 | 의미 | 예시 |
|------|------|------|
| feat | 새로운 기능 | feat: AI 노트 요약 기능 추가 |
| fix | 버그 수정 | fix: 로그인 오류 수정 |
| docs | 문서 관련 | docs: API 명세서 업데이트 |
| chore | 기타 변경 | chore: 불필요한 파일 삭제 |

### PR 작성
- 모든 PR은 기능 브랜치 → `dev` 기준으로 생성  
- PR 제목 형식: `[타입] 간단 설명`  
  - 예: `[Feat] AI 요약 기능 추가`  
  - 예: `[Hotfix] 로그인 오류 수정`
